# 目标网络产品项目介绍

## 1. 一句话摘要

我创造目标网络，是为了把一个人关于职业发展、个人成长、幸福生活的长期愿望，整理成一张可以看见、可以调整、可以复盘、可以持续推进的星图式目标网络。

## 2. 项目起点

我做这个产品的起点，是一个很具体的感受：很多人并不是没有目标，而是没有一个适合长期目标生长的空间。

待办工具擅长提醒今天要做什么，项目管理工具擅长追踪交付节点，笔记工具擅长保存想法。但当问题变成“我想成为什么样的人”“我未来几年要把生活带向哪里”“职业、成长、幸福之间应该如何取舍”时，这些工具都会显得太碎、太平、太急。

我不想再做一个任务列表，也不想做一个让用户每天被红色数字和过期提醒追赶的效率系统。我想做的是一个安静的、自我观察式的产品：用户在深夜、清晨或周末回顾时打开它，看到的不是一堆任务，而是自己的人生目标全景。

所以我把产品命名为“目标网络”。它的底层是结构化数据：目标、父子层级、重要性、进度、支撑关系、依赖关系、冲突关系、行动和复盘记录。它的表层是一张星图：目标不是卡片，而是星体；进度不是冰冷的进度条，而是星核逐渐发光；人生的不同方向不是文件夹，而是可以互相牵引的领域。

从一开始，我就希望它同时满足两件事：

- 它要有足够清晰的数据结构，能承载真实的长期规划。
- 它也要有足够温柔的体验，让用户愿意每周主动回来，而不是被提醒逼回来。

这就是目标网络的起点：我想把长期自我管理从“催促自己完成更多事”，变成“更清楚地看见自己正在成为什么样的人”。

## 3. 商业机会

我看到的商业机会，不在于再做一个更强的待办工具，而在于个人长期目标管理正在出现一个空位。

第一，越来越多的人已经习惯建立自己的知识库、工作流和个人系统。他们会记录灵感、读书笔记、项目计划和生活复盘，但这些内容往往只是被保存下来，并没有真正组织成一个能指导行动的目标网络。信息沉淀很多，方向感却不一定更强。

第二，用户对传统效率工具的心理疲劳越来越明显。很多工具默认人需要被提醒、被催促、被量化、被排名，但长期成长不是冲刺。职业转型、能力建设、健康改善、关系经营、创作积累，都需要一种更低压、更持久、更能承载不确定性的产品形态。

第三，AI 让目标结构化的成本大幅降低。过去，一个人要把模糊愿望拆成目标、成功信号、子目标、行动候选和复盘问题，需要很强的方法论能力。现在，AI 可以在用户已有目标网络的上下文中，帮助补全结构、发现缺口、提出下一步行动。但前提是产品本身要有结构，而不是只提供一个聊天框。

所以我把目标网络定位为“个人长期成长基础设施”，而不是简单的效率工具。它最适合的早期用户，是那些已经有自我管理意识、愿意沉淀个人系统、重视长期复盘的人：知识工作者、创作者、独立开发者、管理者、职业转型者、长期学习者，以及正在认真经营自己生活的人。

这个市场不会只为“多完成几个任务”付费，但会为长期清晰度、行动连续性、个人数据资产和高质量 AI 辅助付费。

## 4. 用户痛点

我在设计目标网络时，主要围绕以下痛点展开。

- 长期目标太抽象。很多人知道自己想成长、转型、变健康、做出作品或改善生活，但不知道如何把愿望拆成有层级、有路径、有验证标准的目标体系。
- 目标之间的关系不可见。现实中的目标不是孤立列表。一个目标可能支撑另一个目标，也可能依赖某个前置条件，还可能和另一个目标争夺时间、精力和注意力。
- 普通进度反馈太机械。百分比、进度条、红绿灯状态适合短期项目，但放在人生目标上容易制造被评价、落后和失败的感觉。
- 本地知识库不够直观。Markdown 文件适合长期沉淀，但文件夹和链接很难直接呈现目标全景、权重分布、推进状态和目标之间的牵引关系。
- 计划、行动、进展和复盘容易断开。用户可能写了很多计划和复盘，但这些内容没有回流到目标结构里，导致记录越来越多，行动路径却越来越模糊。
- AI 建议经常缺少上下文。通用 AI 可以给出建议，但如果不知道当前目标、父目标、子目标、同级目标和已有行动，建议很容易正确但无用。
- 用户担心数据被平台锁住。长期目标和复盘记录是非常私密、长期、重要的数据，用户需要在线体验，也需要可导出、可迁移、可版本化的数据资产。

这些痛点共同指向一个结论：用户需要的不是更多任务，而是一个能把愿望、目标、行动、记录和复盘连接起来的长期系统。

## 5. 产品方案

我把目标网络设计成六层产品能力。

### 5.1 目标网络数据模型

产品的核心不是待办事项，而是目标节点和目标地图。一个用户可以创建多个目标地图，每张地图里都有多层目标节点。每个目标记录标题、状态、周期、领域、父目标、重要性、清晰度、进度、主题色、标签、最后复盘、最近进展和地图坐标。

每个目标都有一组结构化内容：

- 目标摘要：说明这个目标为什么存在。
- 子方向或中期目标：承接上层目标的拆解。
- 成功信号：定义什么才算真实推进。
- 行动候选：把目标连接到可执行动作。
- 复盘问题：帮助用户持续校准目标是否仍然有效。

我还为目标之间设计了横向关系：supports、depends_on、conflicts_with。这样用户看到的不只是树状层级，还能看到哪些目标互相支撑，哪些目标存在依赖，哪些目标正在冲突。

### 5.2 星图式可视化界面

目标网络的核心界面是一张 Goalscape 星图。我没有把目标做成普通卡片，因为卡片会把长期目标变成普通任务。我把目标做成悬浮星体或岛屿，让用户打开产品时，有一种在观察自己内心宇宙的感觉。

职业发展、个人成长、幸福生活是三个基础领域。用户可以在星图中选择目标、拖拽位置、进入子目标分支、返回上级视图，也可以在不同表现模式之间切换。子目标的进度会根据重要性加权聚合到父目标，让上层目标的状态来自真实的下层推进。

我希望这种视觉不是装饰，而是认知工具。用户通过位置、大小、颜色、连接和光芒理解自己的目标结构，而不是只看一列文字。

### 5.3 目标创建与编辑工作流

用户可以创建顶层目标、子目标和同级目标。创建时可以设置领域、周期、重要性、进度、主题色、摘要、成功信号、行动候选和复盘问题。

我刻意把创建流程设计成可深可浅。用户可以只写一个目标名称，先把想法放进网络；也可以展开更多信息，把成功标准、行动和复盘问题一次性写清楚。详情面板则支持快速调整重要性、进度、备注和行动候选。

这个设计背后的原则是：长期目标一开始往往不清楚，产品不应该要求用户一开始就想完整。目标网络应该允许目标从模糊到清晰，从愿望到结构，从结构到行动。

### 5.4 AI 辅助目标推进

我把 AI 助手设计成结构化助手，而不是开放式聊天窗口。它基于当前目标、父链、子目标和同级目标生成建议，并且所有输出都要符合明确的数据结构。

当前 AI 能力包括：

- 优化目标：补全或改写目标定义、成功信号、行动候选和复盘问题。
- 拆解子目标：基于当前目标生成可创建的子目标。
- 分支体检：诊断某个目标分支里的风险、缺口和不一致。
- 本周行动：把目标转化成可以本周执行的行动。
- 目标草稿：在创建目标时自动生成标题、领域、周期、摘要、行动和复盘问题。

我不会让 AI 直接替用户改写一切。AI 生成的是候选建议，用户需要勾选后才会应用。这样既利用了 AI 的结构化能力，也保留了用户对个人目标系统的最终控制权。

### 5.5 记录、周行动与复盘闭环

目标网络不只保存目标，也保存目标推进过程。我把计划、复盘、周复盘、进展日志和周行动都放进同一个产品闭环里。

这个闭环是：

1. 用户定义长期目标。
2. 产品帮助拆成子目标和行动候选。
3. 用户选择本周行动。
4. 用户记录进展、阻碍和学习。
5. 复盘结果反向更新目标状态、进度、关系和下一步行动。

我希望目标网络最终解决的不是“目标怎么写”，而是“目标如何持续推进”。如果一个目标系统不能连接到行动和复盘，它就只是漂亮的愿望清单。

### 5.6 数据同步与可迁移架构

我非常重视数据自主权。长期目标、复盘和个人成长记录不应该被锁在一个封闭平台里。

因此，目标网络同时支持在线应用和 Markdown 数据资产。在线侧使用 Supabase 承载登录、工作区、权限、目标、关系、周行动、记录、审计事件和实时订阅。用户修改目标后，界面可以实时刷新。

本地数据侧使用结构化 Markdown。目标文件保留 frontmatter 和正文结构，可以写回目标文件、周行动文件、计划、复盘和进展记录。同步侧支持把目标网络导出为 Markdown 文件并提交到 GitHub 仓库，从而保留版本历史、备份能力和跨工具迁移空间。

这不是附属功能，而是我对产品信任关系的设计：用户愿意把长期人生规划放进一个产品，前提是产品尊重用户的数据所有权。

## 6. 商业模式

我会把目标网络的商业模式建立在个人订阅上，而不是一开始就做成企业绩效系统。这个产品的核心价值是个人长期清晰度和持续推进力，不应该被过早改造成公司 OKR 或团队考核工具。

我设想的商业分层是：

- 免费层：提供基础目标地图、有限目标数量、基础编辑、手动 Markdown 导入导出，让用户先体验星图式目标管理的核心价值。
- Pro 个人订阅：提供无限目标地图、跨设备同步、历史记录、Markdown/GitHub 同步、高级可视化、AI 目标优化、AI 子目标拆解、AI 周行动建议和更多导出能力。
- AI 增值包：为高频用户提供额外 AI 用量、长上下文分析、分支体检、季度复盘报告和目标重构建议。
- 专业用户版本：面向教练、咨询师、导师和深度个人系统用户，提供多工作区、模板库、客户或学员目标地图、共享复盘空间和只读协作。
- 模板与方法论内容：围绕职业转型、作品集建设、健康习惯、长期学习、创业探索等场景提供目标网络模板，把产品从工具扩展为可复用的成长系统。

我认为用户愿意为三类价值付费：

- 长期清晰度：把模糊愿景变成可观察的目标网络。
- 持续推进力：把目标、周行动、进展和复盘接起来。
- 数据自主权：在线可用，同时保留 Markdown/GitHub 形式的长期资产。

我的商业切入路径会先服务个人高价值用户，建立可展示的目标网络模板和真实使用案例，再向教练、学习社群和小型成长团队扩展。这样产品可以保持“安静、探索、温暖”的核心气质，同时逐步形成更高客单价的专业场景。

## 7. 后续优化方向

接下来，我会围绕“让目标网络更真实地陪伴一个人长期成长”继续优化，而不是简单堆功能。后续重点会放在以下几个方向。

### 7.1 强化目标网络的结构表达

当前产品已经能表达父子层级、重要性、进度和目标关系。下一步我会继续增强目标之间的结构表达，让用户更容易看出哪些目标是核心目标，哪些目标只是阶段性手段，哪些关系正在形成瓶颈。

我希望未来的目标网络不只是“展示目标”，而是能帮助用户回答几个关键问题：

- 哪些目标真正支撑我的长期方向？
- 哪些目标消耗了很多精力，但贡献不大？
- 哪些目标互相冲突，需要取舍？
- 哪些目标长期没有进展，是因为行动不足、定义不清，还是依赖条件没有满足？

这会推动产品从可视化工具，进一步变成个人目标系统的诊断工具。

### 7.2 做深复盘与时间维度

长期目标必须放在时间里看。我后续会加强时间维度，让用户能看到一个目标从提出、拆解、推进、停滞、调整到完成的全过程。

具体方向包括：

- 增加目标历史轨迹，展示重要性、进度、状态和关系的变化。
- 强化周复盘、月度复盘、季度复盘之间的连续性。
- 自动汇总一段时间内的进展、阻碍、行动完成情况和目标变化。
- 帮助用户识别反复出现的问题，例如长期拖延、目标过多、优先级摇摆或行动太大。

我希望复盘不只是写一篇记录，而是让用户真的看见自己如何变化。

### 7.3 提升 AI 的长期上下文能力

现在的 AI 能力主要围绕单个目标或单个分支生成结构化建议。下一步我会让 AI 更理解用户的长期目标网络，而不是只看当前节点。

后续 AI 可以承担更高层次的辅助工作：

- 分析整张目标地图是否过载。
- 发现长期目标和本周行动之间的断裂。
- 根据历史复盘生成季度目标调整建议。
- 帮用户把模糊愿望变成多个可选择的目标路径。
- 在用户新增目标时提醒它可能依赖、支撑或冲突的已有目标。

我的原则仍然不变：AI 只提供候选判断和结构化建议，最终取舍必须留给用户。

### 7.4 优化星图交互和视觉层次

星图是这个产品的核心体验，我会继续打磨它的可读性和操作效率。后续重点不是做更复杂的视觉效果，而是让用户在大规模目标网络里仍然看得懂、找得到、改得快。

优化方向包括：

- 更好的缩放、平移、聚焦和分支下钻体验。
- 更清晰的目标密度控制，避免目标变多后视觉拥挤。
- 更直观地展示支撑、依赖、冲突等不同关系。
- 为不同使用场景提供不同视图，例如人生全景、当前季度、本周行动、停滞目标、冲突目标。
- 继续保持安静、低压、无催促感的视觉语言。

我希望星图既有停下来观看的美感，也有日常管理目标时的效率。

### 7.5 完善数据同步和导入导出

数据自主权会继续是产品底层原则。后续我会让导入、导出、同步和备份更稳定、更透明，让用户知道自己的目标数据在哪里、如何迁移、如何恢复。

优化方向包括：

- 更完整的 Markdown 导入校验和错误提示。
- 更稳定的 GitHub 同步任务状态和失败恢复。
- 支持用户手动触发导出、查看导出历史和回滚版本。
- 为不同用户习惯提供更清晰的数据迁移路径。
- 在同步冲突发生时，提供可理解、可选择的解决方式。

只有当用户相信数据不会被锁住，才会愿意把真正重要的长期规划交给产品。

### 7.6 验证商业化和模板体系

商业化上，我会先验证个人 Pro 订阅是否成立，而不是过早扩展复杂团队功能。目标网络的早期价值应该先在一批高质量个人用户身上被证明。

我会重点验证：

- 哪类用户最愿意持续使用目标网络。
- 用户愿意为什么能力付费，是 AI、同步、历史轨迹、模板，还是高级复盘。
- 哪些目标网络模板最能帮助新用户启动。
- 教练、导师、学习社群是否存在更高价值的专业场景。
- 免费层和 Pro 层之间的边界应该如何设置，既让用户充分体验核心价值，又能支撑产品持续发展。

模板体系会是一个重要方向。我希望未来用户不是从空白开始，而是可以选择职业转型、长期学习、健康改善、作品集建设、创业探索等目标网络模板，再逐步改造成自己的系统。

### 7.7 保持产品气质的一致性

最后，我会持续约束产品气质。目标网络不能因为功能变多，就变成普通效率软件；也不能因为加入 AI，就变成一个自动替用户做决定的系统。

后续每一个优化，我都会用几个问题来判断：

- 它是否让用户更清楚地看见自己的长期方向？
- 它是否让行动和复盘更自然地连接起来？
- 它是否减少焦虑，而不是制造新的压力？
- 它是否尊重用户的数据和最终判断？
- 它是否让用户愿意主动回来，而不是被通知推回来？

如果这些问题的答案是否定的，即使功能看起来有用，我也不会把它作为目标网络的核心方向。
